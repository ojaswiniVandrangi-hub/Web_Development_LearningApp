üî§ HTML Elements and Attributes

Elements are the fundamental building blocks of a webpage. They define the structure, layout, and content of a website. Attributes provide extra information or control the behavior of those elements.

An HTML element consists of a start tag, content, and an end tag. Elements represent different parts of a webpage such as headings, paragraphs, images, links, lists, forms, and much more. There are over 100 standard HTML elements available.

All HTML elements can have attributes. Just like people have specifications such as height, color, or profession, HTML elements can be described or modified using attributes.

Attributes provide additional details about elements. They are always included in the opening tag and usually follow a name="value" pattern.

Real-life Example: Think of a car as an element ‚Äî its color, brand, and model are like attributes that describe the car in more detail.

Example of an element with attributes:
<a href="https://google.com">Visit Google</a>

In the above example, href="https://google.com" is the attribute, where href is the name of the attribute and https://google.com is the value.

üß© Nesting Elements

Nesting means placing elements inside other elements to build a structured web page, just like boxes inside boxes or containers inside containers.

The most basic structure of any HTML page uses nesting. Here's a perfect example: 


<html>
    <head>
        <title>My Webpage</title>
    </head>

    <body>
        <h1>Welcome to my site</h1>
        <p>This is a paragraph.</p>
    </body>
</html>

In the above example:

  • The <html> element is the parent that wraps everything.
  • Inside <html>, we have two nested elements ‚Äî <head> and <body>.
  • Further inside <head>, the <title> is nested.
  • Inside <body>, we nest other elements like <h1> and <p>.

Real-life Example: Think of a tiffin box. The large box holds smaller containers inside it ‚Äî rice in one, curry in another. Similarly, HTML elements sit inside parent elements to stay organized.

Nesting helps maintain structure, hierarchy, and clarity in your webpage. Incorrect nesting can break your layout!

Subscript: Text that appears slightly below the normal line, commonly used in chemical formulas or technical notations.

  • Carbon dioxide ‚Üí CO2
  • Mathematical exponent ‚Üí 52 = 25
  • Chemical expression ‚Üí CH4

5<sup>2</sup> = 25

üìè Inline vs Block Elements (Detailed Explanation)

Inline Elements: These elements stay within the same line of text. They do not force a line break, meaning the content before and after them remains on the same line.

Real-life Example: Imagine highlighting a single word in a sentence without disturbing the sentence flow ‚Äî just like using a highlighter in your book.

Example Code (Inline Element with <span>):

I love <span style="color:red;">HTML</span> and <span style="color:green;">CSS</span> development.

Output:

I love HTML and CSS development.

As you see, the words HTML and CSS are styled but stay in the same line as the rest of the sentence. That's the behavior of Inline elements.

Block Elements: These elements always start on a new line, no matter where you place them in the code. They take up the full available width by default.

Real-life Example: Think of paragraphs or headings. Even if you write them side by side in your code, they will appear one below the other on the webpage.

Example Code (Block Elements):

<p>This is the first paragraph.</p><p>This is the second paragraph.</p>

Output:

This is the first paragraph.

This is the second paragraph.

Even though the two <p> tags are written side by side in the code, the output displays them on separate lines. That's because <p> is a block-level element.

Quick Comparison:

  • Inline Elements: Stay in the same line
  • Block Elements: Start on a new line, take full width

React Introduction

React is an open source library, developed and maintained by the developers of Facebook, Instagram and also a larger community of contributors.

React helps in developing a large scale, single page applications easier.

It is a declarative, efficient and flexible JavaScript library for building UI's

React supports all syntaxes of ECMA script 2015/ES6

React can perform Client Side as well as Server Side Rendering.

Now, I know most of you must got doubt about what is client side rendering? What is server side rendering? Let's look into it deeper

Client Side Rendering:

The browser(client) loads a basic HTML page first and then uses JavaScript to fetch and render the content

Real Life Example:

Imagine you are craving for pizza. We have 2 ways. The first way will be followed by the one who is more health conscious like me. Get all the ingredients from the store like pizza base, cheese, sauces, veggies for topings and make pizza with all the ingredients. Now in this scenario, initially we can see all the ingredients and then after a functionality got added(i.e; baking pizza) the perfect pizza we craved for is just before us. Consider baking pizza as adding javascript to the page.

How it works:

Initially, Browser loads a minimal HTML file with JS

JS framework like React, Angular, Vue etc. will run in the browser, fetch data from APIs and dynamically build the page

This is like loading a web page where your browser builds the content using JS after loading blank page

Live app example: Facebook, Gmail, Netflix

Justification:

When you open Facebook/Gmail you will notice the layout loads first (like the header, sidebar) and actual content like posts, notifications and messages load after a delay

Server Side Rendering:

The server generates the complete HTML content of the page and sends it back to the browser.

Real life Example:

Now, think of the pizza you craved for is store bought. No need to bake. You will get the sight of pizza as you want without delay. This is server side rendering. This is like loading a web page where the server sends full content and see it immediately in the browser.

How it Works:

A request is sent to the server

The server renders the full content with all the data

The browser displays the fully populated page.

Live App Example:

Amazon

Justification:

When you search for a product in Amazon, or open a product page, the entire content is already there in the HTML when the page loads.

Even if we turn off JS, still you can see the product details.

Simple: if you want your page to behave static, go for server side rendering. If you want your page to behave dynamic, go for client side rendering.

Functions:

In React, we can categorize functions in 2 types: 1) Pure functions and 2) Impure functions.

Pure Functions:

The function that returns the same result when the function callback with same argument at multiple times. Hence, it is pure.

function sum(a, b) {
  return a + b;
}
console.log(sum(2 + 3)); // output: 5
console.log(sum(2 + 3)); // output: 5

In both cases, when we pass same arguments into the function, output is same. So, this kind of functions are pure

Impure Functions:

The function may not return the same result when the function callback with same argument at multiple times. Hence, it is impure.

var sum = 0;
function sum(a, b) {
  return sum = sum + value;
}
console.log(sum(2)); // output: 2
console.log(sum(2)); // output: 4

In both cases, when we pass same arguments into the function, output different. So, this kind of functions are impure

Virtual DOM

To create dynamic UIs, relying on traditional DOM wasn't an optimized solution.

Real-life Example:

Imagine you are maintaining a board displaying 10 notes pinned on it. If you want to update just 1 note, the DOM forces you to erase and rewrite all 10 notes again ‚Äî even though only 1 changed. This is inefficient.

In web terms, DOM re-renders the entire list even if one item changes, which is resource-heavy.

This inefficiency is solved using the Virtual DOM.

The Virtual DOM is a lightweight, in-memory representation of the real DOM. When a change occurs, it updates only the changed part ‚Äî not the entire structure ‚Äî and then syncs it with the actual DOM efficiently.

Real-life Analogy:

Consider a painter who maintains a sketch version (Virtual DOM) of their final artwork (Real DOM). If a change is needed, they modify the sketch first, analyze the changes, and only apply what's required on the final artwork ‚Äî saving time and resources.

Used By: Facebook, Apple, PayPal

Comparison Table: DOM vs Virtual DOM

Feature DOM Virtual DOM
Definition Represents the actual structure of the document loaded in the browser. Lightweight JavaScript copy of the real DOM maintained in memory.
Update Speed Slow, updates entire tree. Faster, updates only changed nodes.
Efficiency Low performance on frequent updates. High performance due to selective updates.
Used In Traditional web development. Modern libraries like React.
Rendering Direct manipulation of real DOM. Changes applied in memory first, then synced.

Features of React

1. Components

React applications are built using components. A component is an independent, reusable block of UI, like a Lego brick. Each component can have its own logic and appearance.

Real-life Analogy: A car built from parts (components). Wheels, engine, seats can be reused.

Types of Components:

  • Functional Components ‚Äì functions that return JSX
  • Class Components ‚Äì ES6 classes with state and lifecycle methods (less common now)
function Welcome(props) {
  return <h1>Hello, {props.name}!</h1>;
}

2. JSX (JavaScript XML)

JSX is a syntax extension that allows you to write HTML-like code within JavaScript.

Analogy: Chocolate chips (HTML) in cookie dough (JavaScript).

// With JSX
const element = <h1>Welcome to React!</h1>;
// Without JSX
const element = React.createElement('h1', null, 'Welcome to React!');

3. Transpiling

JSX cannot be understood by browsers directly. Tools like Babel are used to convert JSX into standard JavaScript ‚Äî this process is called transpiling.

Example:

// JSX
const greeting = <h1>Hello, world!</h1>;

// Transpiled JS
const greeting = React.createElement('h1', null, 'Hello, world!');

4. Abstract Syntax Tree (AST)

During transpilation, JSX is first parsed into an Abstract Syntax Tree (AST). AST is a tree representation of your code‚Äôs syntax.

Babel uses AST to understand code structure and apply transformations.

Analogy: Think of AST as a blueprint of a building. Architects (Babel) can modify it before real construction (JS code execution).
